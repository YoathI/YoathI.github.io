FusionANNS

《Towards High-throughput and Low-latency Billionscale Vector Search via CPU/GPU Collaborative Filtering and Re-ranking》
总结

本文提出 FusionANNS：一种面向十亿规模 ANNS 的“CPU + GPU”协同处理架构。FusionANNS 仅使用一块入门级 GPU，即可同时实现高吞吐、低延迟、成本高效与高准确性。其核心思想是通过 CPU/GPU 协同的过滤与重排序，尽可能减少 GPU、CPU 与 SSD 之间的数据搬移。

FusionANNS概述

本文提出了FusionANNS，一个基于SSD的GPU加速的ANNS系统，能够在十亿规模数据集上同时实现高吞吐、低延迟、成本高效与高准确性：

1、针对GPU/CPU协同过滤，使用多层级索引，显著降低CPU和GPU之间的数据传输。

2、使用启发式重排序，在重排序过程中消除不必要的I/O计算。

3、基于优化存储布局提出冗余感知的I/O去重，以进一步提高I/O效率。

研究背景

在CPU/GPU协同加速的ANNS系统中，为构建高质量IVF索引，多数系统将边界向量复制到相邻的倒排表中，这会使索引规模显著增大，即使使用PQ压缩，GPU的HBM仍无法容纳全部压缩索引，导致GPU与CPU之间进行频繁的数据置换。由于PQ会带来精度损失，需要搭配向量重排序以提升准确性，但在给定准确性约束下，很难为每个查询确定最小重排序所需向量数量。原始向量远小于SSD的最小读取粒度（4KB），请求原始向量往往会引发显著的读放大，导致低I/O效率的重排序。

相关技术

PQ与重排序技术re-ranking

PQ技术用来对高维向量进行压缩，设包含N个向量的数据集采用PQ压缩：首先将每个向量均匀划分为M个子空间，每个子空间包含N个子向量。随后，对这些子向量进行聚类以生成码本，码本由各簇的质心集合构成。借助码本，每个子向量都可以用其最近的质心来近似表示。每个子空间的聚类数通常设为256，从而每个簇 ID 可用 1 字节表示。当所有码本生成后，每个向量即可被压缩为一个M字节的PQ码。在处理查询时，首先生成距离查找表，其中包含每个子查询向量与对应子空间内各质心之间的距离。查询向量q与压缩向量v的近似距离可表示为：

其中，$M$表示子空间总数，$q_i$表示第 iii 个子查询向量，$c_i(v_i)$表示压缩向量在第 iii 个子空间对应的质心。因此，$\operatorname{dist}(q,v)$可通过以PQ码为地址查表快速获得。最后，将所有$\operatorname{dist}(q,v)$累加即可得到q与v的近似距离。

PQ是有损压缩，经过PQ技术计算距离后得到的top-k排序是近似的，因此在高精度要求下需要搭配重排序技术，对PQ快速筛选得到的top-L候选（L>k）使用它们的原始向量进行更精确的距离计算并重新排序，得到top-k结果。

HBM与GPU显存

HBM即高带宽内存，可以作为GPU的显存。除此之外，GPU的显存也可以用GDDR6/GDDR6X等实现。

主要挑战

challenge 1

将分层索引、乘积量化与GPU加速技术简单组合，会导致比仅采用分层索引的SPANN更高的延迟和更低的吞吐量。因为即使采用PQ技术进行向量压缩，GPU的HBM仍无法容纳所有倒排表，导致GPU与CPU之间频繁的数据传输，造成新的性能瓶颈。

challenge 2

由于PQ是有损压缩，为达到相同精度的查询水平，需要对GPU生成的中间结果进行重排序。参与重排序的候选向量n通常是top-k向量的数倍。但不同查询所需的最小重排序数量差异很大，仅对少数拖尾查询增加重排序数量才有实际效益。若为所有查询统一设置固定的重排序数量，将不可避免地导致大量不必要的 I/O 操作与距离计算。因此需要为每个查询动态确定其最小重排序向量数量。

challenge 3

重排序过程需要从SSD上读取大量原始向量数据，从而引入额外的I/O请求。由于单个向量与I/O操作最小单元差距过大，会导致显著的读放大问题，造成重排序阶段极低的I/O效率。参与重排序的向量之间高度相似，可以通过重新组织SSD上的数据布局来缓解读放大问题。

FusionANNS具体设计

FusionANNS的具体设计包括多层级索引、CPU/GPU协同过滤、启发式重排序和冗余感知I/O去重机制等。

多层级索引

FusionANNS索引设计为多层级索引结构，采用图索引+IVF索引，GPU HBM常驻PQ压缩向量，内存中存放导航图和倒排表中的向量IDs，SSD上存放原始向量，索引结构如下所示：

FusionANNS的多层级索引结构

FusionANNS采用分层平衡聚类算法对数据集进行划分，生成的倒排表包含多个向量ID和对应向量内容，倒排表数量为数据集中向量总数的约10%。FusionANNS采用与SPANN相同的复制机制以解决边界向量问题。

主存中的索引：基于倒排表的质心，FusionANNS使用SPTAG构建图索引，在查询时快速查找与查询向量最接近的前m个倒排表。FusionANNS仅提取每个倒排列表中的向量ID作为元数据（不包含向量内容）并存储于主存中。当图索引和元数据构建完成后，中间生成的完整倒排列表即可丢弃。

该图索引通过向空图中持续插入新向量的方式构建。当一个向量作为新顶点加入时，会创建新的边，将其与前 k 个（通常为 64 个）最近邻连接。随后，其邻居节点更新其最近邻集合，以限制边的最大数量。

GPU HBM中的压缩向量： FusionANNS将所有PQ压缩向量存储在GPU的HBM中，以避免GPU和CPU之间频繁的数据交换问题。在FusionANNS通过导航图获取所有候选向量后，仅将这些向量ID发送到GPU进行距离计算。

SSD上的原始向量：FusionANNS仅在SSD上存储原始向量，用于重排序阶段。由于原始向量的数据量约为倒排列表的1/8，这样显著降低了存储空间开销。

GPU/CPU协同过滤

对于向量查询过程，FusionANNS采用GPU/CPU协同过滤，CPU遍历内存中的导航图，获取top-m倒排表，然后将对应列表中的向量IDs传输给GPU；GPU对其进行距离计算，并返回top-n向量IDs给CPU；CPU从SSD读取对应的原始向量进行重排序并返回最终top-k结果。具体的查询流程如下：

FusionANNS架构和查询流程

FusionANNS查询流程：

  1. 当接收到一个查询向量时，FusionANNS首先由GPU生成该查询向量的距离查询表，用于后续PQ距离计算。

  2. 同时，CPU遍历主存中的导航图，查找与查询向量最接近的m个倒排表。

  3. 然后，CPU查询元数据，获取候选倒排表中的所有向量IDs。

  4. 接着，CPU将这些向量IDs传输到GPU，并调用GPU内核进行进一步处理。

  5. 当GPU接收到向量IDs后，首先通过并行哈希模块对其进行去重。

  6. 对于每个向量IDs，GPU从HBM中读取对应的PQ向量，并计算与其查询向量之间的PQ距离。

    在该过程中，GPU为PQ向量的每个子空间分配一个线程，用于访问距离查找表中的对应预计算值；随后由协调线程对各子空间距离进行累加，得到该候选向量的PQ距离

  7. 然后，GPU按距离升序对所有候选向量进行排序，并将前n个向量的IDs返回CPU。

  8. 由于基于PQ的中间结果存在一定误差，CPU根据这些向量ID从SSD中读取对应的原始向量进行重排序。

  9. 最终，CPU输出前k个最近邻结果。

启发式重排序

对于由GPU处理得到的候选向量，需要进行向量重排序，以达到相同的查询精度。若采用固定的重排序向量数量，可能会导致不必要的I/O操作和距离计算开销或精度下降。

为此，FusionANNS采用一种启发式重排序机制，以最小化I/O操作和距离计算开销。其核心为：在保证高精度的前提下，保守地设置一个相对较大的重排序上限；一旦后续搜索对提升查询精度不再具有明显收益，立即终止重排序过程。

FusionANNS将重排序过程划分为多个小批次并顺序执行。每个小批次包含相同数量的候选向量。由于候选向量已按距离升序排列，较早执行的小批次更有可能包含最终前 k 个最近邻中的向量。当完成一个小批次后，使用一个轻量级反馈控制模型判断后续小批次是否仍有助于提升查询精度。

基本机制

为简化问题，FusionANNS使用一个优先队列 Q（最大堆）维护当前的前 k 个最近邻。初始时最大堆为空。

  1. 计算该批次中每个向量与查询向量之间的精确距离；
  2. 若某向量距离小于当前最大堆中的最大值，则将其插入堆中；

FusionANNS启发式重排序算法

首先初始化最大堆 $Q$ 为空，并使用 StabilityCounter 来记录变化率持续低于 $\epsilon$ 的次数。Tasks 的大小表示需要进行批内重排序的候选向量总数，BatchSize 表示每个小批次中包含的候选向量数量。

对于每个小批次，在执行任务前从 $Q$ 中获取当前 Top-k 向量的 ID。随后，依次执行各项任务，包括从 SSD 读取原始向量、计算其与查询向量的距离，并将距离小于最大堆中最大值的向量插入 $Q$ 中。当小批次处理完成后，从 $Q$ 中提取已更新的 Top-k 向量 ID，并计算两次连续小批次之间 $Q$ 的变化率 $\Delta$。若 $\Delta < \epsilon$，则 StabilityCounter 加 1；一旦 StabilityCounter 超过阈值 $\beta$，则终止重排序过程。否则，重置 StabilityCounter 并继续处理后续批次。最后，返回 $Q$ 中的最终 Top-k 向量。

冗余感知I/O去重

FusionANNS 在重排序阶段需要从 SSD 上读取原始向量，以对 GPU 返回的中间结果进行精细化重排序。但原始向量远小于页面粒度，针对这些原始向量的单次请求往往会导致严重的读取放大。并且，由于重排序过程引入了大量的随机小 I/O 操作，I/O 延迟对端到端查询延迟有着很大影响。为此，FusionANNS优化数据布局以改善SSD上的空间局部性，并利用冗余感知I/O去重机制来缓解读放大影响。

优化存储布局

在离线创建内存索引时：

  1. 对于导航图中的每个质心，使用一个存储桶（bucket）来存储若干个最接近该质心的原始向量，并保证各桶之间不存在重复向量。
  2. 对于每个桶，若其尺寸无法与 SSD 页面对齐，则采用max-min算法，根据未对齐部分的尺寸对桶进行合并。
  3. 最后，将所有桶组合成单个文件存储在 SSD 上，并使用内存中的一张表来维护向量与 SSD 页面之间的映射关系。

小批次内与跨小批次 I/O 去重

基于优化后的数据布局，FusionANNS设计了两种 I/O 去重机制：合并小批次内映射到同一SSD页面的I/O请求和利用DRAM缓冲区消除后续小批次中的冗余 I/O。具体如图所示：

FusionANNS数据布局和I/O去重

假设重排序包含两个 mini-batch：

  • mini-batch 0：V2、V4、V6
  • mini-batch 1：V5、V8、V9

在 mini-batch 0 执行时：

  • 系统查询映射表，获得对应的 SSD 页 ID；
  • 若 V2 与 V6 位于同一页 P0，则合并为一次 I/O；
  • 若 P0 与 P2 尚未存在于 DRAM 缓冲区，则通过两次 I/O 将其加载至 DRAM；

在 mini-batch 1 执行时：

  • V5、V8、V9 位于不同 SSD 页；
  • 若 DRAM 缓冲区中已包含 P2（其中含有 V5），则无需再次读取；
  • 仅需对 P1 与 P3 发起两次 I/O 请求。

通过该机制，系统显著减少了 SSD 的 I/O 次数。
