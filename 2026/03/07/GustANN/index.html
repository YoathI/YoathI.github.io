GustANN概述

现有的基于SSD的图索引ANNS系统通常依赖CPU执行搜索，但CPU计算能力成为系统瓶颈，导致无法充分利用多SSD的带宽。为解决这一问题，论文提出了以GPU为中心，CPU协同辅助的系统架构，即由 GPU 负责主要的图遍历与向量计算，而 CPU 辅助数据传输与调度，从而构建系统 GustANN：

  1. 内存高效的 GPU 内核：通过最小化图搜索过程中的显存占用，提升GPU与SSD并发度。
  2. CPU 辅助数据传输：有效缓解了 GPU 端在数据传输时受到的 PCIe 带宽瓶颈。
  3. 枢纽搜索：实现了跨 SSD 负载均衡。

研究背景

为保证成本效益，对于十亿级向量的高吞吐量检索，采用SSD比内存更佳，因其单位存储成本更低且容量更大；在高吞吐量ANNS场景中，图索引相较其他类型更具成本优势，因为图结构ANNS通过把向量连接成近邻图，搜索时只需在图中访问少量节点即可找到近邻，在同样硬件条件下可以处理更多查询，从而实现更高的系统吞吐量。

在高吞吐场景下CPU的整体算力存在根本性不足，无法充分利用多SSD的理论最大带宽，造成现有基于SSD的CPU图ANNS索引因CPU瓶颈难以实现高吞吐量。

  • 以往的研究聚焦低延迟场景：当 CPU 发出一个读取请求后，必须等待 SSD 返回数据，导致SSD读取时间大于CPU计算时间，使得I/O延迟成为基于磁盘ANNS的性能瓶颈。
  • 在高吞吐场景（同时处理大量查询、发起大量SSD I/O）中：着重点在于QPS，由于CPU计算能力有限，其处理数据的速度可能低于 SSD 的数据传输能力，无法充分利用SSD的理论最大带宽，造成CPU算力成为性能瓶颈。

因此，在ANNS系统中引入GPU算力成为可行方向，但现有基于GPU的ANNS方案仍存在局限：

  • CPU主导、GPU辅助架构：由CPU执行图遍历，将向量计算卸载到GPU。ANNS的过程中仍受CPU性能制约。
  • 纯GPU架构：将图遍历和向量计算全量迁移到GPU以充分利用其大规模并行能力。受限于GPU显存容量与数据传入带宽，需要占用大量显存来维护visited table等与查询相关的数据及结构，且纯GPU架构通过GPU内核直接发起SSD读取，会形成新的性能瓶颈。

针对上述问题，本文提出了GustANN，采用以GPU为中心、CPU协同辅助的架构：由GPU负责图遍历计算，并 利用CPU协同处理数据传输， 避免CPU搜索瓶颈对性能的限制，并通过在CPU侧进行I/O管理以缓解GPU入站带宽瓶颈问题。

相关技术

基于磁盘的图结构ANNS

图被存储为多条记录，分布在 SSD 页上，每条记录对应一个图节点。每条记录包含全精度向量和由邻居 ID 构成的邻居列表。节点邻居数量由对应参数控制，确保记录大小固定，同一记录不会跨页存储。

基于磁盘的图结构存储布局

对应 ANNS 搜索流程算法如下：

候选集 S：存储当前查找到的距离最近的 L 个候选节点（ L 为搜索长度）；结果集 E：记录候选节点与查询向量的精确距离。

搜索从固定的入口节点P开始，进行多轮迭代：

  1. 读取当前待搜索节点u所在的SSD页面，获取对应全精度向量和邻居列表；
  2. 将u标记为已搜索，计算与查询向量的距离，将u插入结果集；
  3. 遍历u的所有邻居，为减少全精度向量的读取次数，通过PQ距离估算后插入候选集S；
  4. 裁剪候选集规模至L以控制迭代次数，选择距离查询向量最近的未探索节点作为新的待搜索节点；
  5. 当候选集未空时，搜索终止，返回结果集中top-k个节点。

Root Complex

根复合体（Root Complex, RC）是 PCIe 体系结构中的核心控制组件，负责连接 CPU/内存子系统与 PCIe 设备，如 SSD、GPU 等。

主要挑战

Challenge 1： 高内存占用的搜索过程限制了系统并发度，而 GPU 与 SSD 均依赖高并发设计。

GPU 和 SSD 均以高并发为设计核心：GPU 通过大量并行线程实现高算力，SSD 则通过并行处理多个请求以实现高带宽。因此结合 GPU 与 SSD 需要高度并发的系统设计。GPU 显存有限，若存储visited table等用于查询的数据结构，会导致 GPU 的显存被大量占用，同时处理的查询（并发查询）数量变少，从而无法充分发挥 GPU 和 SSD 的并行能力。

Challenge 2： 如何有效利用 CPU 协助数据传输以缓解 GPU 侧带宽瓶颈，尚未得到充分探索。

在图索引ANNS场景下采用GPU直接访问技术，难以获得理想性能。所有 PCIe 设备首先连接至root complex，再通过高速链路连接至 CPU 与 DRAM，设备之间通常不存在直接连接。这导致SSD与GPU间传输带宽不对称——随着SSD数量增加，其聚合带宽可能超过GPU入站带宽上限，形成新瓶颈。

Challenge 3： 在 RAID-0 布局下，多 SSD 之间存在负载不均衡，降低整体搜索性能。

RAID-0 将数据按固定大小的条带切分，然后依次分布到多个磁盘（或 SSD）上，以提升读写带宽和并行性能。

为最大化利用SSD带宽，即使使用 RAID-0 布局把数据均匀分布到多块 SSD，上层的查询仍然会出现“延迟不均衡”，从而拖慢 GPU 批处理搜索的速度。因为不同查询访问的SSD不同，如果某块SSD被多次访问，会使多SSD之间负载不均衡，可能造成显著的长尾延迟。由于GPU通常是批处理查询，长尾延迟会拖慢整个批次的执行。

GustANN具体设计

GustANN 采用以 GPU 为中心、CPU 协同的架构，以实现高吞吐和高性价比的 ANNS。其整体架构如下：

  • 针对挑战1，GustANN 设计了一种内存高效的图遍历 GPU 内核，通过冗余计算换取更小的内存空间，并设计了一种适用于 GPU 的并行图搜索算法来降低冗余计算的开销。
  • 针对挑战2，GustANN利用CPU辅助数据的选择与传输：访问图节点时，首先将对应页面从SSD传输至CPU DRAM，以充分利用SSD聚合带宽；随后仅将目标节点从DRAM传输至GPU，从而节省有限的GPU入站带宽。
  • 针对挑战3，GustANN 采用 pivot search 将查询分散至图的不同区域，从而避免单一区域的微尺度负载不均衡。

数据布局

GustANN 采用基于磁盘的图结构ANNS将向量及其邻居信息存储在 SSD 上，并采用 RAID-0 方式对多块 SSD 进行数据条带化存储。对于 pivot search，GustANN 在 GPU 显存中存储一个 pivot 图。同时，将大多数数据结构如PQ压缩向量从 DRAM 移到 GPU 显存。

GustANN 中 DRAM 的主要用途是作为 CPU 辅助传输中的缓冲区，以实现从 SSD 到 GPU 的选择性传输。与 GPU 直接访问相比，该方法能够缓解 GPU PCIe 带宽瓶颈。GustANN 主要利用 DRAM 的带宽优势，且其容量需求远低于其他基于 DRAM 的十亿级 ANNS 系统。

向量搜索

单向量查询

初始化：GustANN 以与 DiskANN 类似的方式初始化查询。此外，还会执行枢轴搜索以实现负载均衡和决定主搜索的入口节点。

搜索：搜索过程由 CPU 主导，CPU 负责协调 SSD 读取和 GPU 上的搜索。搜索过程采用与基于磁盘的图结构ANNS类似的方案，并将大多数计算放到 GPU 上执行。搜索的迭代过程包括：

  1. SSD 阶段：从 SSD 读取所选探索节点 u 对应的 SSD 页面到 DRAM，然后将页面中的目标记录传输到 GPU。
  2. GPU 阶段：在 GPU 上执行距离计算和数据结构更新，并选择下一次 SSD 阶段的探索节点。

多向量查询

为了支持高吞吐量的 ANNS，GustANN 将查询放入批处理中，以进一步利用 GPU 的计算能力和 SSD 的读取带宽。

但是由于图遍历过程是顺序的，在不同阶段之间存在强依赖关系，例如探索节点以及来自 SSD 的记录。因此，这两个阶段无法同时运行，导致一半的硬件资源被浪费。为了进一步挖掘硬件潜力，GustANN 引入了查询间流水线（inter-query pipelining）。不同于将所有查询放入一个大型批处理中，GustANN将其拆分为多个小批次（mini-batches）。这些批次彼此独立，因此两个阶段可以被不同的小批次填充，从而使所有硬件资源都能够得到利用。

> 如 t1 时间，SSD 阶段执行 batch1，t2 时间，SSD阶段执行 batch2，GPU阶段执行 batch1。

GPU内核设计

visited table 内存占用过高

论文通过分析说明在图索引搜索过程中所维护的 visited table 占据了较大的内存比例。相比之下，候选集仅需存储 L 个节点；其他结构（如记录缓存和 PQ 查找表）在整个搜索过程中大小固定。且GustANN 需要支持高并发，由于GPU显存有限，且需存储PQ编码等数据结构，会导致开销过大，不可接受。

GPU上的冗余计算是可接受的

在 CPU 上执行 ANNS 时，通常在距离计算前消除重复邻居，以避免 PQ 距离的冗余计算。然而，GPU 线程以 warp 为执行粒度，同一warp 内线程始终执行相同指令或处于等待状态。因此，单个线程的冗余计算往往可被其他线程掩盖，不会显著增加总体时间开销。

> 即传统基于 CPU 的 ANNS 算法通常依赖占用大量内存的数据结构，以空间换时间的方式消除冗余计算，但这种方式并不适用于 GPU。而GPU 的 SIMT架构能够天然容忍一定程度的冗余计算。

基于上述观察，GustANN 首先执行距离计算与排序，再对排序后的数组进行去重。由于排序后的数组可通过线性扫描完成去重，因此无需访问记录表等额外数据结构。此外，GPU 的 warp 执行机制能够掩盖距离计算与排序中的冗余计算开销，不会带来显著时间开销。

算法设计

GustANN 对基于磁盘的图索引 ANNS 进行改进，采用“先排序、后去重”的搜索算法。在搜索过程中，根据PQ距离将候选集组织为有序数组，数组中的每个元素包含候选节点 ID 及其 PQ 距离，并额外使用 1 个比特标记该节点是否已被探索。算法执行流程如下：

在每一轮搜索迭代中，首先执行 SSD 阶段读取当前探索节点的所有邻居；随后通过下列步骤在 GPU 阶段将这些邻居加入候选集：

1、PQ距离计算：计算当前探索节点的所有邻居的 PQ 距离。

使用 R（邻居数量）个 GPU 线程，每个线程负责计算一个邻居的 PQ 距离。

由于 GPU 以 warp 为执行单位，重复邻居导致的冗余计算可以被其他线程的计算隐藏，因此不会显著增加整体时间开销。

2、排序：将所有邻居追加至候选集，并按照 PQ 距离进行排序：

  1. 对当前轮新增的邻居进行排序；
  2. 将排序后的邻居与上一轮剩余的候选集部分进行归并。

两个步骤均采用并行归并排序，排序阶段同样使用 R 个 GPU 线程，并将其与 PQ 距离计算内核融合，以更好地利用 GPU 的高带宽共享内存。

CPU辅助数据传输

SSD 不共享 PCIe通道，多 SSD 带宽聚合较高，但 GPU 带宽受到 PCIe 通道限制，导致SSD 与 GPU 之间的带宽不对称，造成GPU 对 SSD 的直接访问无法充分利用多个 SSD 的带宽潜力。

搜索过程中的GPU阶段，GPU线程会读取一个驻留在 SSD 上的记录，其大小小于 SSD 页。但 GPU 直接访问无法选择性传输数据，因此利用 CPU 来辅助数据进行传输，并使用 CPU 的 DRAM 作为中转缓存区，如上图所示。步骤如下：

  1. 将 SSD 页从 SSD 传输到 DRAM；
  2. 仅将正在探索节点的图记录从 DRAM 传输到 GPU。

枢纽搜索

图索引ANNS采用单入口设计会导致在前几轮的迭代过程中，所有查询只会访问图中入口节点附近的区域。对于同一查询集合，节点被访问次数差异过大，可能导致某些 SSD 被多次请求，而其他 SSD 处于空闲状态，导致多 SSD 负载不均衡现象，造成尾延迟问题。

> 单入口设计会在图遍历的初始阶段仅访问入口节点附近的局部区域，而该区域中的节点在不同 SSD 之间分布不均。由于所有查询都必须首先访问该区域，这种微观层面的不均衡被进一步放大。

GustANN使用 pivot search 将查询分散到图的不同区域，而不是集中在入口节点附近的单一区域：

  1. 缓解负载不均衡问题，每个区域包含的节点更少，因此访问次数的差异会减小。
  2. 如果初始区域更接近查询的最近邻，则可以获得更快的搜索速度。

GustANN 引入GPU 驻留 pivot 图来缓解由单入口设计导致的 SSD 负载不均衡问题：

  • 从数据集中随机选取 pivot 向量（如对数据集采样 0.1%）来表示不同区域，并在 GPU 内存中构建一个 GPU 驻留的 pivot 图 G′。对于一个查询，首先在 G′ 上执行 top-1 搜索（即 pivot search）。得到的结果（采样集合中的最近向量）将被用作 SSD 驻留图搜索的入口节点。

Pivot search 与上述图搜索类似，但存在差异：

  1. 更短的搜索长度（即候选集长度），以减少整体搜索开销。
  2. 使用全精度向量，而不是压缩的 PQ 编码，因为 pivot 图占用的内存很小。
  3. 使用 选择排序而不是归并排序。因为搜索长度较小，选择排序更容易实现并行化。

其他

Warp(CUDA)

Warp 是 NVIDIA GPU 架构中线程调度和执行的基本单元，固定由 32 个线程组成。执行模式为SIMT， 同一个 Warp 内的所有 32 个线程在同一时刻执行完全相同的指令，但它们处理的是不同的数据。

如果 32 个线程中，部分线程执行 A 指令，另一部分线程执行 B 指令（例如遇到 if-else 分支），GPU 硬件会通过“分支掩码”（Branch Mask）进行序列化处理：先执行完 A 分支的线程挂起等待，再执行 B 分支。这种现象称为 分支发散 (Branch Divergence)。

Kernel Fusion

Kernel Fusion 即内核融合。Kernel 是一个被设计成在 GPU 等设备上运行的函数，比如矩阵相乘、向量加法、归一化等，是并行处理的最小逻辑单元。Kernel Fusion 是把多个小的计算任务合并成一个更大的任务，一次性计算完成，而不是一个一个单独调用。Kernel Fusion的优势：

  • 减少内存读写：多个操作分开做时，中间结果需要读写内存，速度慢。融合后中间结果不用写内存，直接用，速度更快。
  • 减少内核启动和调度开销：每次 kernel 启动都需要 CPU 提交任务并由 GPU 进行执行调度，因此会产生启动与调度开销。Kernel fusion 通过减少 kernel 启动次数，从而降低这些开销。
  • 提升并行度和硬件利用率：融合后的 kernel 通常具有更大的计算规模，可以更充分地利用 GPU 的计算单元和内存带宽，从而提升整体性能。
